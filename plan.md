# 订阅刷新与 Outlook 启动联动实施计划

1. 在后端新增“刷新全部订阅来源”后台任务，统一管理全部订阅 URL 的刷新状态、进度与结果。
2. 将订阅刷新粒度改为“单个 URL”：
   - 每个 URL 独立处理。
   - 每个 URL 最长等待 3 分钟用于“拉取并解析出至少 1 个节点”。
   - 若 3 分钟内仍未拉取到任何节点，或 fetch/parse 失败，则判定该 URL 刷新失败。
3. 对单个 URL 刷新失败时：
   - 找到该 URL 当前关联的全部旧节点。
   - 若其中有节点在节点池中，则先从实际监听配置移除并 reload。
   - 将这些旧节点统一转入失败节点池。
   - 失败节点清空端口、延迟、国家信息，并写入刷新失败原因。
4. 对单个 URL 拉取成功时：
   - 使用新节点替换该 URL 的旧节点。
   - 启动后续测速、测国家、自动入池流程。
   - 该 URL 的刷新阶段以“成功拉取到节点”为成功判据，测速/入池结果单独统计。
5. 新增刷新任务查询 API，返回：
   - 全部 URL 总数、已完成数、成功数、失败数。
   - 每个 URL 的当前状态、节点数、测速/入池进度与错误信息。
   - 整体任务是否已完成。
6. 修改 `start_easy_outlook.ps1`：
   - 启动 easy_proxies。
   - 调用“刷新全部订阅来源” API。
   - 持续轮询直到所有 URL 都进入终态。
   - 完成后根据最终 `/api/nodes/pool` 数量更新 `D:\Outlook\OutlookRegister\config.json` 的 `proxy.port_end`。
   - 刷新任务进入终态后，不论整体状态是 finished 还是 failed，都继续读取真实节点池。
   - 若最终节点池为空则直接报错，不启动 Outlook。
   - 否则再启动 `python D:\Outlook\OutlookRegister\main.py`。
7. 验证：
   - 运行 Go 测试 / 构建，确保后端 API 与脚本无语法错误。
   - 用本地 dummy Outlook 配置验证脚本流程。
   - 验证日志输出、进度轮询、端口更新逻辑正确。
